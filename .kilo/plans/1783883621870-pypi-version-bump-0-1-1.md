# Plan: Bump dot-ai-evals 0.1.0 → 0.1.1 and republish to PyPI

## Context

Publishing `dot_ai_evals-0.1.0-py3-none-any.whl` to PyPI failed with HTTP 400:

> File already exists ('dot_ai_evals-0.1.0-py3-none-any.whl', with blake2_256 hash 'd69e820562523d8ae317e894c8f7c8bbe3a9feb35c70ed89b90b97cd550af2b3').

PyPI forbids filename reuse: once a file is uploaded, that exact filename can never be re-uploaded, even after deletion. The fix is to publish under a new version number.

## Decisions

- **Version bump:** patch `0.1.0` → `0.1.1` (re-publish of same code, no functional changes).
- **Build/publish tooling:** `hatch build` + `hatch publish` (matches existing hatchling build backend).
- **Stale artifacts:** clean `dist/` before rebuilding to avoid re-attempting the 0.1.0 upload.
- **Out of scope:** git commit / tag (only do if explicitly requested).

## Steps

1. Edit `pyproject.toml:7` — change `version = "0.1.0"` to `version = "0.1.1"`.
2. Clean build artifacts: remove everything in `dist/` (stale 0.1.0 wheel + sdist).
3. Build: run `hatch build`.
4. Verify `dist/` contains only `dot_ai_evals-0.1.1-*` files (no 0.1.0 artifacts remain).
5. Publish: run `hatch publish`.
   - Requires PyPI credentials in env: `HATCH_INDEX_USER` (or `__token__`) and `HATCH_INDEX_AUTH` (API token), or equivalent configured in `~/.pypirc`.
6. Sanity check: `pip install dot-ai-evals==0.1.1` succeeds and imports cleanly.

## Risks / Notes

- 0.1.1 will publish the *current* code as a patch over the already-released 0.1.0. Acceptable for a re-publish; if code has drifted significantly, consider a minor bump instead.
- If `hatch publish` is unavailable, fall back to `python -m build` + `twine upload dist/dot_ai_evals-0.1.1-*`.
- Do not commit or tag unless the user explicitly asks.

## Validation

- `dist/` listing shows only 0.1.1 artifacts.
- `hatch publish` returns success (no 400 / filename-reuse error).
- `pip install dot-ai-evals==0.1.1` resolves from PyPI and the package imports.
