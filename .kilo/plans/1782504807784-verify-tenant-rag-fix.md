# Plan: Verify tenant-RAG misclassification fix reproduces correctly

## Context

The user pasted a `rubrics.yaml` generated by `ai-eval init` for a "Tenant wise RAG
pipeline" that came out wrong:

- `project_type: chat`
- both tasks (`single_eval_chain`, `chat_messages_chain`) typed `chat`
- only the `hallucination_rate` metric on each

That output is the exact bug addressed by two existing (already-implemented,
uncommitted) plans:

- `.kilo/plans/fix-tenant-rag-misclassification.md` — broadened the LangChain
  retriever heuristic (call-site detection + provider-package imports) and
  hardened `_classify_project_type`.
- `.kilo/plans/pgvector-detector.md` — added `PGVectorDetector`.

Both plans mark first-class multi-tenant modeling (tenant_id input, per-tenant
golden sets) as out of scope; this verification does **not** change that.

## Findings / follow-ups (no source changes proposed here)

1. **The pasted rubrics is stale.** It predates the two fixes. Re-running
   `ai-eval init --force` on the user's real tenant-RAG repo will regenerate
   `project_type: rag` with RAG metrics. No further detector work is required
   for this shape.
2. **CLI `--dry-run` UX gap (observation, not a defect in the fix):**
   `ai-eval init --dry-run` has no global `--cwd` option, so it scans the
   process cwd only. Running it against an external repo requires `cd`-ing
   into it first. When invoked from the `ai-eval` tool's own repo it reports
   `0 AI task(s)` because the tool repo isn't a target AI project. If the user
   hit this, a follow-up could add a `--cwd`/`--root` global option — but that
   is orthogonal to the misclassification fix and is **out of scope** for this
   verification.
3. **Tenant scoping still absent from the rubric.** The fix ensures tenant-RAG
   is *detected as RAG*, but `tenant_id` is not surfaced as an input field and
   there is no per-tenant golden-set support. This is the documented out-of-scope
   follow-up from `fix-tenant-rag-misclassification.md` and remains open.
